The insight — two different jobs
Middleware and Server Components have different capabilities and should have different jobs. 
MIDDLEWARE (Edge Runtime)
  Runs in V8 isolate — no Node.js APIs
  Cannot do async database/API calls
  Can only read request cookies synchronously
  Purpose: lightweight routing gate

SERVER COMPONENT (Node.js Runtime)
  Full Node.js — can call any async function
  Can read and WRITE cookies
  Can call refreshAccessToken()
  Purpose: actual auth logic + data fetching
What middleware should check
// WRONG — what was causing the bug
const token = req.cookies.get('access_token')?.value;
// access_token expires every hour → user logged out every hour

// CORRECT — standard approach
const hasSession = Boolean(
  req.cookies.get('refresh_token')?.value
);
// refresh_token = "session exists" (valid for 30 days)
// access_token freshness checked in Server Component
How Auth0, Clerk, Vercel do it
All major auth providers use this same two-layer pattern:

Layer 1 — Middleware: Does the session exist? (session cookie / refresh token)
Layer 2 — Server Component: Is the access token fresh? Refresh if not.

This is documented in Next.js official docs under "Authentication Patterns".

EDGE src/middleware.ts
Checks refresh_token only — not access_token. Middleware is a session gate, not a token validator. Access token refresh handled in Server Components.
import { NextRequest, NextResponse } from 'next/server';
import { COOKIE_KEYS, PUBLIC_ROUTES, BYPASS_PREFIXES, ROUTES } from '@/lib/constants';

// ─────────────────────────────────────────────────────────────
// MIDDLEWARE — Edge Runtime
//
// STANDARD APPROACH (how Auth0, Clerk, Vercel all work):
//
// Middleware is a LIGHTWEIGHT GATE only.
// It answers ONE question: does this user have a session at all?
//
// It checks refresh_token (not access_token) because:
//   - access_token expires frequently (1hr)
//   - refresh_token is the actual session indicator (30 days)
//   - Checking access_token here causes logout on expiry
//   - Server Components handle access_token refresh properly
//
// WHO DOES WHAT:
//   Middleware     → session exists? (refresh_token check)
//   Server Layout  → access_token valid? refresh if not
//   useAuthQuery   → API call 401? refresh and retry
//   /api/auth/*    → token operations themselves
//
// This removes all the complexity of silent-refresh pages,
// race conditions, and request type detection.
// ─────────────────────────────────────────────────────────────

export default function middleware(req: NextRequest) {
  const { pathname } = req.nextUrl;

  // ── Always bypass — never intercept ────────────────────────
  if (BYPASS_PREFIXES.some(p => pathname.startsWith(p))) {
    return NextResponse.next();
  }

  const isPublicRoute = PUBLIC_ROUTES.some(
    r => pathname === r || pathname.startsWith(r)
  );

  // ── Check session existence via refresh_token ───────────────
  // refresh_token = "user has an active session"
  // access_token  = "user's current token is fresh"
  // We only care about session existence here.
  // Access token freshness is handled in Server Components.
  const hasSession = Boolean(
    req.cookies.get(COOKIE_KEYS.REFRESH_TOKEN)?.value
  );

  // No session + protected route → login
  if (!hasSession && !isPublicRoute) {
    const url = new URL(ROUTES.LOGIN, req.url);
    url.searchParams.set('callbackUrl', pathname);
    return NextResponse.redirect(url);
  }

  // Has session + hitting login/register → dashboard
  if (hasSession && isPublicRoute) {
    return NextResponse.redirect(new URL(ROUTES.DASHBOARD, req.url));
  }

  // All other cases → let through
  // Server Components will handle access_token refresh
  return NextResponse.next();
}

export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico|.*\\.png$|.*\\.jpg$|.*\\.svg$).*)',
  ],
};
SERVER src/lib/auth/requireAuth.ts
NEW — Single helper called in every protected layout. Handles valid token, expired token (inline refresh), and no session. Replaces all silent-refresh complexity.
import 'server-only';

// ─────────────────────────────────────────────────────────────
// requireAuth — Server-Side Auth Helper
//
// Called at the top of every protected Server Component layout.
// Handles access_token refresh transparently.
//
// This is the standard pattern used by:
//   - Next.js official docs
//   - Vercel's own auth examples
//   - Auth0 Next.js SDK internals
//
// FLOW:
//   1. Has access_token? → return user, render page
//   2. No access_token, has refresh_token?
//      → refreshAccessToken() inline
//      → new cookies written
//      → return user, render page
//   3. Neither?
//      → redirect to /login
//      (middleware should have caught this but belt + braces)
// ─────────────────────────────────────────────────────────────

import { redirect }           from 'next/navigation';
import { getAccessToken, getSessionUser } from '@/modules/auth/lib/cookies';
import { refreshAccessToken }  from '@/modules/auth/lib/refresh';
import { ROUTES }             from '@/lib/constants';
import type { SessionUser }   from '@/types/auth.types';

interface AuthResult {
  user:        SessionUser | null;
  accessToken: string;
}

export async function requireAuth(): Promise<AuthResult> {
  let accessToken = await getAccessToken();

  // access_token missing but middleware let us through
  // means refresh_token exists → try refresh inline
  if (!accessToken) {
    const newToken = await refreshAccessToken();

    if (!newToken) {
      // Refresh also failed — belt and braces redirect
      redirect(`${ROUTES.LOGIN}?reason=expired`);
    }

    accessToken = newToken;
  }

  // Decode user from stored idToken — no extra API call
  const user = await getSessionUser();

  return { user, accessToken };
}
SERVER src/app/(dashboard)/layout.tsx
One line: await requireAuth(). Handles all auth scenarios. No silent-refresh redirect needed.
// src/app/(dashboard)/layout.tsx
// SERVER COMPONENT — no 'use client'
//
// ─────────────────────────────────────────────────────────────
// PROTECTED DASHBOARD LAYOUT
//
// Uses requireAuth() which handles:
//   - access_token valid     → continue
//   - access_token expired   → refresh inline → continue
//   - refresh_token expired  → redirect /login
//
// No silent-refresh page needed.
// No middleware complexity.
// This is the standard Next.js App Router auth pattern.
// ─────────────────────────────────────────────────────────────

import { requireAuth }     from '@/lib/auth/requireAuth';
import { Providers }       from '@/providers/Providers';
import { TopBar }          from '@/components/layout/TopBar';
import { ToastContainer }  from '@/store/ToastContainer';

export default async function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  // Single call handles ALL auth scenarios:
  //   valid token     → returns user
  //   expired token   → refreshes silently → returns user
  //   no session      → redirects to /login
  const { user } = await requireAuth();

  return (
    <Providers>
      <div className="flex flex-col min-h-screen bg-gray-50">
        <TopBar user={user} />
        <div className="flex-1 flex flex-col">
          {children}
        </div>
        <ToastContainer />
      </div>
    </Providers>
  );
}

await requireAuth() at top. Token silently refreshed before data fetching if needed.
// src/app/(dashboard)/locations/[locationId]/layout.tsx
// SERVER COMPONENT — no 'use client'

import { requireAuth }          from '@/lib/auth/requireAuth';
import { getLocationsTree, getLocationConfig } from '@/modules/locations/graphql/server-locations';
import { LocationSidebar }      from '@/modules/locations/components/LocationSidebar';

interface Props {
  children:  React.ReactNode;
  params:    Promise<{ locationId: string }>;
}

export default async function LocationLayout({ children, params }: Props) {
  const { locationId } = await params;

  // requireAuth handles expired token silently — no extra code
  await requireAuth();

  // Parallel data fetch — both fire at the same time
  const [tree, config] = await Promise.all([
    getLocationsTree(),
    getLocationConfig(locationId),
  ]);

  return (
    <div className="flex flex-1 h-[calc(100vh-56px)]">
      <LocationSidebar
        activeLocationId={locationId}
        initialTree={tree}
        initialConfig={config}
      />
      <main className="flex-1 overflow-y-auto bg-gray-50">
        {children}
      </main>
    </div>
  );
}
